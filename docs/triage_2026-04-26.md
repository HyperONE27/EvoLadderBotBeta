## Bugs — needs investigation, no design doc yet

### 2v2 leaderboard doesn't update before 2v2 result resolves — ⏰ pending

**Effort:** M — investigation + targeted fix.

**Hypothesis:** event ordering between `_apply_match_resolution` (2v2) and the `leaderboard_updated` broadcast. The 1v1 path may broadcast leaderboard before MMR write completes, but 2v2 doesn't, or vice versa. Compare `_match.py:_apply_match_resolution` against `_match_2v2.py:_apply_match_resolution_2v2` for ordering of: cache update → DB write → MMR recompute → leaderboard rebuild → broadcast.

**Where to start:** `backend/orchestrator/transitions/_match_2v2.py` and search for `leaderboard_updated` and `rebuild_leaderboard_2v2` calls.

### 2v2 race verification false-positive when teams swapped — ⏰ pending

**Effort:** M — bug in replay verifier.

**Hypothesis:** `replay_verifier.py` is comparing races positionally (team 1 player 1 race == replay player A race) when it should be comparing per-team as a *set* — order within a team should not matter. Fix: in the 2v2 path, sort each team's expected races and the replay's per-team races before comparing, or use `Counter`.

**Where to start:** `backend/algorithms/replay_verifier.py`, look for the 2v2 race-check function.

### Heartbeat embed update vs cancel queue embed update can race — ⏰ pending

**Effort:** M — concurrency.

**Hypothesis:** the searching-message edit loop and the user-initiated cancel both call `message.edit(...)` without coordination. If cancel arrives mid-heartbeat, the heartbeat edit can clobber the cancel embed.

**Fix shape:** per-uid asyncio.Lock around the edit, or check `cache.active_searching_messages.get(uid)` before each heartbeat edit and skip if cleared. Don't introduce a global lock — that serialises all heartbeats.

**Where to start:** wherever the heartbeat task is defined (likely `bot/services/` or inside the queue command flow).

### Map "clumpiness" — Golden Wall too often — ⏰ pending

**Effort:** S-M — depends on whether we just want logging or an actual fix.

**Current behavior:** `backend/algorithms/match_params.py` and `match_params_2v2.py` use `random.choice` over the non-vetoed map pool. If the pool size after vetoes is small, Golden Wall (or any single map) will naturally cluster.

**Two paths:**
1. **Diagnostic only.** Add a `/admin map_distribution` command that reports map-frequency over the last N matches. Confirm the bias is real before changing the algorithm.
2. **Fix the bias.** Bias-correct `random.choice` with a recent-map memory: track the last 5-10 maps assigned globally and downweight them in the next pick. Cheap; no schema changes if the memory is in-process.

Recommendation: do #1 first.
