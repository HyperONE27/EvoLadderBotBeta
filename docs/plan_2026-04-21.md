## Shared repo context (read before implementing any task)

These are not rules — `CLAUDE.md` is the rule book — but they are non-obvious gotchas that have bitten this codebase before. The implementer should weigh each task against them.

- **Writethrough is mandatory and order-sensitive.** Every state mutation goes through `TransitionManager`. Inside a transition: write to Supabase via `DatabaseWriter` *first*, then update the `StateManager` DataFrame. Reversing this order silently desyncs the cache from the DB on a partial failure. Several tasks below add new transitions — do not invert this.
- **All in-memory state lives in one process.** Backend runs `--workers 1` intentionally. Any new background task, scheduler, or in-memory dict (e.g. the deferred-ping tasks for activity notifier) is per-process and lost on restart. That's fine for ephemeral state, but persistent things must go through Supabase.
- **No raw datetimes anywhere on the wire.** Supabase rejects raw `datetime` objects. Use `common/datetime_helpers.utc_now()` and `to_iso()` (or the existing `_serialise_event_row` helper) before any DB call or JSON serialization. Polars columns are `Datetime("us", "utc")`.
- **DataFrame mutations only via `TransitionManager`.** Don't reach into `StateManager` and `df.with_columns(...)` from a lookup or endpoint. Add a transition method.
- **Bot ↔ backend events flow over WebSocket only.** Any new state transition that the bot needs to react to (e.g. activity notifier wants to know about queue joins; partner-confirmation needs to know when leader queues) requires both a backend `ConnectionManager.broadcast(...)` call and a `bot/core/ws_listener.py` handler.
- **Game-mode routing on WS payloads.** 2v2 broadcasts include `"game_mode": "2v2"`. 1v1 broadcasts omit the field. The bot dispatches on `data.get("game_mode", "1v1")`. New events should follow this.
- **Discord rate limits + the message queue.** Non-interaction Discord calls go through `bot/core/message_queue.py` (40 msgs/sec, two-tier priority). Interaction responses (`response`, `followup`, `edit_original_response`) bypass the queue and have their own 3-second budget. The activity status embed *edits* are non-interaction calls — they must go through the queue or you'll hit 429s.
- **Locale parity is enforced.** Any new key added to `data/locales/enUS.json` must be added to all 5 other files (`base`, `esMX`, `koKR`, `ruRU`, `zhCN`) in the same lexicographic position. CI doesn't check this; reviewers will.
- **Naming conventions.** Never bare `_uid` — always `_discord_uid`. Never bare `_name` for a player — `_player_name` or `_discord_username`. Applies to schemas, TypedDicts, ephemeral types, and JSON payloads.
- **Static data lookup is duplicated by design.** Both `StateManager` (backend) and `Cache` (bot) implement `StaticDataSource`. New static JSON (e.g. `time_zones.json`) must load via `common/loader.py` `JSONLoader` so both processes get it.
- **`check_if_banned` has a side effect.** It populates `Cache.player_presets` and `Cache.player_locales`. Several command flows rely on this side effect to skip a backend round-trip. If a new command flow doesn't go through `check_if_banned`, it must fetch these explicitly.
- **`on_ready` fires on every reconnect.** Module-level guards prevent re-initializing the aiohttp session, message queue, and command tree. Any new startup logic (e.g. activity-status channel discovery) must respect this guard or it will run repeatedly.
- **Replay parsing runs in a `ProcessPoolExecutor`.** No async, no asyncio imports, no event-loop access in `replay_parser.py`. The pool can die from a sc2reader segfault — `Backend.ensure_pool_healthy()` resurrects it. New replay-related work that lives in the parser process must respect this.
- **Channel manager is optional.** If `CHANNEL_MANAGER_URL` is unset, channel creation is silently skipped. New code that *depends* on the chat history (task 5) needs to handle the "no channel was ever created" case gracefully.
- **Cache-before-DB in `_apply_match_resolution` is intentional.** The comment at `backend/orchestrator/transitions/_match.py:607` explains why `count_game_stats` needs the cached match before MMR is recomputed. Don't "fix" this without understanding it; the retry wrapper for Cloudflare 5xx (already specced) is the right shape.

### 1. Replays dashboard for content creators (hidden command)

**Goal.** Let trusted casters search the replay library by filters (length, race, map, MMR band, etc.) without exposing the feature to normal users.

**Decisions.**
- **Not** a separate microservice. Lives in the existing bot to reuse `aiohttp` session, `Cache`, `MessageQueue`.
- **Not** a slash command — invisible to non-casters.
- **Trigger.** `on_message` keyword. Exact word `replays` (no prefix, no slash) sent in a DM to the bot. Case-insensitive match.
- **Module.** `bot/commands/secret/replay_command.py`. Mirrors the structure of existing user commands but exposes a `handle_replays_message(bot, message)` entry point instead of registering with `tree`.
- **Routing.** New branch in `bot/core/app.py` `on_message` (alongside the existing replay-upload and new-user branches): if `message.content.strip().lower() == "replays"`, gate by `is_content_creator` lookup, then call into `replay_command`.
- **Authorization.** New table `content_creators` (Supabase + StateManager DataFrame). Columns: `discord_uid` (PK), `display_name`, `granted_at`, `granted_by_discord_uid`, `is_active`. Owner-only `/owner caster {discord_uid} {add|remove}` to manage entries. Non-listed users hitting the keyword see no response (treat as if the keyword wasn't recognized).
- **UI surface.** Initial reply is a search panel: dropdowns for `game_mode` (1v1/2v2), `race_matchup`, `map`, `min_length_minutes`, `max_length_minutes`, `mmr_band`. Submit button runs the query and edits the message into a paginated results embed with replay download links (Supabase Storage signed URLs).
- **Backend endpoint.** `POST /caster/replays/search` — gated by an internal token header (or by validating the discord_uid against `content_creators` server-side). Returns `[{match_id, players, races, map, length, mmr_avg, replay_url, played_at}]`.

### 2. Persistent activity status channel + join/leave broadcasts

**Goal.** Make queue activity glanceable in a public channel so players don't have to DM-subscribe to know when others are online.

**Decisions.**
- **Single embed message** in a designated channel, edited in-place. Survives bot redeploys.
- **Channel discovery.** New env var `ACTIVITY_STATUS_CHANNEL_ID`. On startup, the bot fetches the channel, scans recent messages for one authored by itself, and reuses its `message.id`. If none exists, creates one. Persisted nowhere; rediscovered each startup.
- **Embed contents.** Live counts of players in 1v1 / 2v2 queues, count of active matches, "last queue join: 3m ago" markers, optional small bar of last-hour activity.
- **Update cadence.** Edit on every queue-join / queue-leave / match-found / match-completed event (subscribe to the same WS events the bot already consumes). Throttle edits to no more than once every 5s to stay under Discord rate limits.
- **Plus** anonymous public-channel broadcasts on join and leave events (separate messages, low-priority message queue). Anonymous = "Someone joined the 1v1 queue" / "Someone left after 2 minutes". This is *in addition to* the live embed.

**Files to touch.**
- `bot/core/config.py` — read `ACTIVITY_STATUS_CHANNEL_ID`.
- `bot/core/bootstrap.py` — `Cache.activity_status_message_id`, populated on `on_ready`.
- `bot/core/app.py` (or new `bot/services/activity_status.py`) — startup discovery/create logic; throttled edit loop.
- `bot/core/ws_listener.py` — already routes events; just add hooks for the activity status updater.
- `bot/components/embeds.py` — `ActivityStatusEmbed` builder.
- Locale files — strings for the embed and the broadcast messages.

**Open spot-check items.**
- Throttle interval (5s default — okay or different?).
- Whether the public broadcasts go to the *same* channel as the live embed or a separate one.
- Whether match-found/match-completed should also broadcast (might be too noisy).

**Repo context worth considering.**
- The bot reconnects to Discord frequently in normal operation (`on_ready` fires repeatedly). The status-channel discovery routine must be idempotent — fetch and reuse, never blindly create a second message.
- `MATCH_LOG_CHANNEL_ID` is the existing pattern for "channel ID via env var"; copy that wiring style (validation in `bot/core/config.py`, fail loud at startup if invalid).
- Edits at high frequency hit Discord's per-channel rate limit independent of the bot-wide 40 msgs/sec budget. The 5s throttle should be measured per-message, not just per-broadcast batch.
- The bot already has live `Cache.leaderboard_1v1` / `leaderboard_2v2` populated via WS push; the status embed should read from in-memory cache rather than re-querying the backend on every edit.
- If broadcasts share the activity channel with the live embed, edits to the embed will fall behind broadcast messages in the channel. This is a UX call — Discord doesn't pin edited messages to the top.

### 3. Activity ping commitment delay + cooldown wakeup

**Goal.** Stop pinging notify-subscribers when the queuer immediately leaves; also re-ping subscribers whose cooldown elapsed while someone is still queued.

**Decisions.** Hybrid approach.
- **Deferred ping on join.** When a `queue_join` event is observed, schedule an asyncio task that fires after `QUEUE_NOTIFY_COMMITMENT_SECONDS` (default 60). On `queue_leave` or `match_found` for that uid before the timer fires, cancel the task. Only fires if the queuer has actually waited the threshold.
- **Periodic sweep for cooldown wakeups.** Every 30s, scan current queues. For each subscriber whose cooldown has elapsed *and* who has not been pinged for the currently-queued cohort, send a ping. This handles "Alice was queued for 10 minutes alone; Bob's cooldown elapsed mid-wait, but no new queue_join fires to wake him up".

**Files to touch.**
- `common/config.py` — `QUEUE_NOTIFY_COMMITMENT_SECONDS = 60` and re-export from `bot/core/config.py`.
- `bot/core/ws_listener.py` (or wherever notify pings dispatch today) — replace immediate dispatch with deferred-task pattern; add cancellation on leave/match.
- New module `bot/services/activity_notifier.py` — owns the deferred-task dict, the periodic sweep, the in-memory `last_sent` map.

**Open spot-check items.**
- Confirm 60s threshold (TODO suggests 60s; could be 30s or 90s).
- Sweep interval — 30s reasonable?
- What counts as a "currently-queued cohort" for sweep dedup — anyone in queue right now, or specifically the ones who joined since last sweep?

**Repo context worth considering.**
- Match wave-tick is at the top of every minute. A 60s commitment threshold means a queuer who joins at `:55` and hits the next wave at `:00` could be matched *before* the commitment fires — that's fine (the ping was no longer needed) but the deferred task must check "is this user still queued and not in a match" before sending.
- The deferred tasks dict is in-memory only. Bot restart drops all pending pings — acceptable per the design backlog at the bottom of `TODO_beta.md` (`/notifyme` cooldown enforcement is also in-memory by design).
- Notifications opt-in/cooldown lives on the `notifications` table per the design backlog. The sweep should read this from the in-memory `Cache` (or backend lookups) rather than DB on every tick.
- `queue_join_activity` is the existing WS event that subscribers receive. Consider whether the new commitment-deferred ping should reuse this event or be a new event type — reusing risks confusing existing subscriber semantics.

### 4. Time zones + quiet hours in `/setup` and `/notifications`

**Goal.** Let users set a local time zone and quiet hours so notifications respect their sleep.

**Decisions.**
- **DST handling library.** Python stdlib `zoneinfo` (built into 3.14, no new dependency). Each curated TZ entry stores an internal IANA name for DST resolution but does not expose it to users.
- **Region structure.** TZ options grouped by the existing geographic regions in `data/core/regions.json`. User picks region first (region they already chose during setup pre-fills), then time zone within that region.
- **Storage.** New columns on `players` (or `notifications`) table:
  - `time_zone` — string (one of the curated entries' codes, e.g. `US_PACIFIC`)
  - `quiet_hours_start_local` — int 0–23 (hour of day in their local time)
  - `quiet_hours_end_local` — int 0–23
  - Nullable; null means "no quiet hours configured".
- **Quiet hours enforcement.** When the activity-notifier is about to ping a user, convert "now" into their local time via `zoneinfo`, check if it falls in `[quiet_hours_start_local, quiet_hours_end_local)` (with wraparound for e.g. 22→6), suppress if so. No queueing; the ping is just dropped.
- **New JSON file** `data/core/time_zones.json` — curated list keyed by geo region. Draft below for spot-check.

**Draft `data/core/time_zones.json` (for spot-check).** One entry per timezone slot. `tz_code` is the persisted value; `iana` is internal-only for DST math; `display` is what users see (will be localized via locale files separately).

## Appendix: Matchmaker exploration (first-principles tradeoffs)

Captured for the deferred matchmaker migration so re-derivation isn't needed next time.

**The original symptom (as written in TODO).** Wave-based matching causes "synchronization issues" where the same players keep getting paired against each other.

**Why this happens with the current 60s wave.** Two players who queue in the same minute always get evaluated together. If they're the closest MMR match in that wave, they pair. If they finish the match and re-queue at roughly the same cadence, they land in the same wave again. Repeat indefinitely. Other players who could break the pattern would have to queue at a *different* cadence to land in a different wave.

**Design axes to consider.**
1. **Eventing model**: wave (current) vs continuous (run on every queue event) vs hybrid (faster waves, e.g. 10s).
2. **Recent-pairs penalty**: do we add a fitness term that penalizes recently-played pairs? If so, how much weight vs the existing MMR-diff² term, and how long is "recent"?
3. **MMR tolerance growth**: should waiting expand acceptable MMR difference? (Current matchmaker has `exponential_wait_bonus` for *preferring* waits, not for tolerating bad matches.)
4. **Queue floor**: should the matchmaker refuse to pair if the only candidate is a recent opponent and queue is small (i.e. wait for a third party to break the cycle)?

**Tensions called out in the original TODO.**
- Don't make two willing players wait 5–6 minutes to play each other when they're the only ones online. (Argues against strict recent-pairs penalty.)
- But do prefer new matchups when available. (Argues for a *soft* recent-pairs penalty.)
- MMR-diff vs novelty tradeoff: how much MMR mismatch is worth the variety?

**Cheapest experiment.** Add a recent-pairs soft penalty to the existing wave matchmaker (no architectural change), then measure repeat-pair rate before/after with the existing event log. If the symptom resolves, no migration needed. If it doesn't, then revisit eventing model.

**Verdict for now.** Defer. Owner should re-confirm the symptom is still happening in pre-beta before any code work.
