## 1. Uniform Before/After State Warnings on All Admin Commands

**Goal:** Every admin command shows a structured `BEFORE тЖТ AFTER` diff for all changed fields in the success embed, following a consistent pattern.

### Current State

| Command | Preview | Success |
|---------|---------|---------|
| `/admin ban` | `BanPreviewEmbed` тАФ shows target name/UID | `BanSuccessEmbed` тАФ shows new ban state only |
| `/admin resolve` | `ResolvePreviewEmbed` тАФ shows result to be set | `AdminResolutionEmbed` тАФ shows before/after MMR тЬЕ |
| `/admin statusreset` | `StatusResetPreviewEmbed` тАФ shows target | `StatusResetSuccessEmbed` тАФ shows old status only |

`resolve` is closest to the target pattern. `ban` and `statusreset` need work.

### Plan

1. **Define a standard "admin diff" embed layout** used by all admin success embeds:
   - Title: action name (e.g. "Ban Toggled", "Status Reset")
   - Field per changed attribute: `field_name: old_value тЖТ new_value`
   - Footer: admin who performed the action + timestamp

2. **`/admin ban` тАФ `BanSuccessEmbed`** (`bot/components/embeds.py:2694`):
   - Add `old_is_banned: bool` parameter.
   - Display: `Ban Status: тЭМ Not Banned тЖТ тЫФ Banned` (or vice versa).
   - Pass the old value from the command flow тАФ the player dict is already fetched in the command handler before the toggle.

3. **`/admin statusreset` тАФ `StatusResetSuccessEmbed`** (`bot/components/embeds.py:4103`):
   - Already receives `old_status`. Add `new_status` (always `"idle"`) explicitly.
   - Display: `Status: queueing тЖТ idle` or `Status: in_match тЖТ idle`.
   - Optionally include whether any queue entry or active match was cleaned up.

4. **`/admin resolve`** тАФ already shows before/after MMR. Extend to also show:
   - `Match Result: conflict тЖТ player_1_win` (the resolution chosen).
   - `Match Status: conflict тЖТ resolved`.

5. **Owner commands (`/owner admin`, `/owner mmr`):**
   - `/owner admin` (`ToggleAdminSuccessEmbed`): show `Role: inactive тЖТ admin` or `admin тЖТ inactive`.
   - `/owner mmr` (`SetMMRSuccessEmbed`): show `MMR ({race}): 1500 тЖТ 1650`.

### Files to Modify

- `bot/components/embeds.py` тАФ update success embed classes
- `bot/components/views.py` тАФ pass old-state values through confirm callbacks
- `bot/commands/admin/ban_command.py` тАФ capture old ban state before toggle

### Complexity

Low-medium. Mostly embed formatting changes + threading old values through the confirm flow. No backend changes needed тАФ old values are already available from the initial player/admin lookup.

## 2. Admin Match-Info Resend Command (Diagnostic + Resend)

**Goal:** New `/admin resend {match_id}` subcommand that shows delivery diagnostic info and allows resending `MatchInfoEmbed` to players who didn't receive it.

### Current State

- `ws_listener.py:296-402` (`_on_both_confirmed`) sends `MatchInfoEmbed1v1` to both players on the `both_confirmed` WebSocket event. Failures are logged but not surfaced to admins.
- `cache.active_match_messages` tracks which players received the message.
- `/admin match {match_id}` shows match details but has no resend capability.

### Plan

1. **New admin subcommand: `/admin resend {match_id}`** in a new file `bot/commands/admin/resend_command.py`.

2. **Diagnostic phase:**
   - Fetch match data from backend (`GET /matches_1v1/{match_id}` or `GET /matches_2v2/{match_id}`).
   - Check `cache.active_match_messages` for each player UID in the match.
   - Build a `ResendDiagnosticEmbed` showing:
     - Match ID, status, players
     - Per-player: тЬЕ Delivered / тЭМ Not in cache (may not have received it)
   - Attach a `ResendConfirmView` with per-player "Resend" buttons + "Resend All" button.

3. **Resend phase:**
   - On button press, re-fetch match data from backend to get current state.
   - Build fresh `MatchInfoEmbed1v1` (or 2v2) + `LobbyGuideEmbed`.
   - Send via high-priority message queue (same path as `_on_both_confirmed`).
   - Update `cache.active_match_messages` with new message reference.
   - Edit the diagnostic embed to reflect successful resend.

4. **Guard rails:**
   - Only allow resend for matches in `confirmed` status (not completed/aborted/abandoned).
   - Rate-limit: one resend per player per match to prevent spam.

### Files to Create/Modify

- **Create:** `bot/commands/admin/resend_command.py`
- **Modify:** `bot/components/embeds.py` тАФ add `ResendDiagnosticEmbed`
- **Modify:** `bot/components/views.py` тАФ add `ResendConfirmView`

- **Modify:** `bot/commands/admin/__init__.py` тАФ export registration function

### Complexity

Medium. The resend logic mirrors `_on_both_confirmed` but triggered manually. Main risk: stale cache state if bot restarted since match creation (cache is in-memory only). The diagnostic should note this limitation.

## 3. Block USB/FER Players from Matching Against Americas-Server Opponents

**Goal:** The matchmaker should never pair USB or FER players with opponents whose resolved server would be on the Americas.

### Current State

- Server resolution is purely table-driven (`match_params.py:51-67`, `cross_table.json`).
- The matchmaker (`matchmaker.py`) has no awareness of player regions тАФ it only sees MMR, races, and wait time.
- Region lookup happens *after* matching, in `resolve_match_params`.
- USB and FER are in `regions.json` with `game_region: "AS"`.

### Plan

**Approach: Post-match filter in the orchestrator, not in the matchmaker.**

The matchmaker is stateless and pure тАФ it operates on `QueueEntry1v1` which has no region data. Rather than polluting the matchmaker with geographic awareness, filter at the orchestrator level:

1. **Add a region-compatibility check** in a new helper, e.g. `backend/algorithms/region_filter.py`:
   - Input: two player region codes + cross_table.
   - Output: `bool` тАФ whether the resolved server is acceptable.
   - "Unacceptable" = the resolved server is in a blocklist. Start with `{"USW", "USC", "USE"}` (Americas servers). Make this configurable via `backend/core/config.py`.

2. **Where to apply the filter** тАФ two options:

   **Option A (recommended): Pre-filter matchmaker input.** In `orchestrator.run_matchmaking_wave()`, before calling the pure matchmaker function, remove queue entries where the player's region is in a "cannot match Americas" set (`{"USB", "FER"}`), *unless* all remaining players are also in that set. This avoids USB/FER being paired with NA players entirely.

   Actually this is too blunt тАФ USB/FER players should still match EU and Asia players. The real constraint is: "if USB/FER is paired with a player whose resolved server would be Americas, reject that pairing."

   **Option B (better): Post-filter matchmaker output.** After `run_matchmaking_wave()` returns match candidates, check each candidate's resolved server. If it falls in the blocklist, reject the match and return both players to the remaining queue. This is simpler and doesn't require modifying the matchmaker at all.

3. **Implementation (Option B):**
   - In `backend/orchestrator/orchestrator.py`, after calling `matchmaker.run_matchmaking_wave()`:
     - For each `MatchCandidate1v1`, look up both players' regions.
     - Call `_resolve_server()` to get the server.
     - If server is in the Americas blocklist, remove from candidates, add both players back to remaining queue.
   - The rejected players stay in queue with incremented `wait_cycles` and try again next wave.

4. **Config:**
   - `BLOCKED_REGION_SERVER_PAIRS` in `backend/core/config.py`: e.g. `{("USB", "Americas"), ("FER", "Americas")}` or more simply a set of `(region, server_set)` pairs.

### Files to Modify

- `backend/core/config.py` тАФ add blocked region/server config
- `backend/orchestrator/orchestrator.py` тАФ add post-match server validation
- `backend/algorithms/match_params.py` тАФ optionally expose `_resolve_server` as public

### Complexity

Low. A few lines of post-filtering logic. The tricky part is deciding the right abstraction so it's extensible to other region restrictions later. Should also apply the same logic to the 2v2 matchmaker path.

## 5. Fix Replay Parsing Producing False Draws (Match #2163)

**Goal:** Replays where `replay.winner` is `None` and no defeat messages exist should not silently resolve as draws.

### Current State

`replay_parser.py:157-180`:
```python
winner = replay.winner
if winner is None:
    # Fallback: look for "was defeated!" messages
    remaining = {p.name for p in replay.players}
    for msg in replay.messages:
        if msg.text.endswith("was defeated!"):
            remaining.discard(msg.text.replace(" was defeated!", ""))
    if len(remaining) == 1:
        winner_name = next(iter(remaining))
        ...

if winner is None:
    result_int = 0  # draw
```

If both fallbacks fail (no `replay.winner`, no defeat messages), `result_int = 0` тЖТ draw. This is incorrect тАФ the result is **indeterminate**, not a draw.

### Plan

1. **Distinguish "draw" from "indeterminate"** in the parser output:
   - `result_int = 0` should remain for genuine draws (if they exist in SC2/Evo).
   - Add a new result: `result_int = -1` or a separate field `"indeterminate": True` when winner cannot be determined.
   - Map to `"indeterminate"` in `_RESULT_INT_TO_STR_1V1`.

2. **Update the verifier** (`replay_verifier.py`):
   - 2v2 already handles `"indeterminate"` (line 154-158) тАФ it blocks auto-reporting.
   - Apply the same logic to 1v1: if parsed result is `"indeterminate"`, do NOT auto-resolve. Players must report manually, or admin resolves.

3. **Update the bot's report flow:**
   - If replay parse returns indeterminate, show a message like "Replay could not determine the winner. Please report the result manually."
   - The report dropdown should still be available (not locked behind replay validation for this case).

4. **Fallback improvement** тАФ also try `PlayerLeaveEvent` as a third heuristic:
   - If exactly one player has a `PlayerLeaveEvent` before the other, that player likely lost.
   - This catches the case described in the TODO: "someone leaves the game and their replay somehow does not capture this action."

### Files to Modify

- `backend/algorithms/replay_parser.py` тАФ change draw fallback to indeterminate, add PlayerLeaveEvent heuristic
- `backend/algorithms/replay_verifier.py` тАФ handle indeterminate for 1v1
- `bot/core/ws_listener.py` or `bot/components/views.py` тАФ surface indeterminate result to players

### Complexity

Medium. The parser change is small, but the downstream effects (verifier, bot UI, auto-resolve logic) need careful testing. Should unit-test with a replay file that reproduces the #2163 scenario if possible.

## 6. Migrate Wave-Based Matchmaking to Non-Wave / Faster Algorithm

**Goal:** Eliminate synchronization issues where players repeatedly match the same opponents across waves. Move to either event-driven matching or faster waves with no obligation to match within a single wave.

### Current State

- Matchmaker runs every 60 seconds (`app.py:26-51`), synchronized to the top of each minute.
- `wait_cycles` increments by 1 per wave, expanding MMR window.
- Hungarian algorithm finds optimal bipartite matching per wave.
- **Problem:** With small player pools, the same two players keep matching because there's no penalty for repeat pairings. If player A and B are the only ones in queue, they match every wave.

### Plan

This is the largest item. Two sub-approaches, in order of increasing complexity:

### Phase 1: Quick Win тАФ Add Recent-Opponent Penalty (Keep Waves)

Keep the 60-second wave architecture but add a **recent opponent penalty** to the scoring function.

1. **Track recent matches** тАФ add a lightweight data structure:
   - `recent_matches: dict[tuple[int, int], datetime]` in `StateManager` тАФ maps `(uid_a, uid_b)` (sorted) to the timestamp of their last match.
   - Populated when a match is created, pruned on access (entries older than `RECENT_MATCH_PENALTY_WINDOW` removed).
   - Config: `RECENT_MATCH_PENALTY_WINDOW` (e.g. 30 minutes), `RECENT_MATCH_PENALTY` (e.g. 500 MMR-equivalent penalty).

2. **Modify scoring** in `matchmaker.py`:
   - `_build_candidates()` computes `score = mmr_diff┬▓ - wait_bonus`.
   - Add: `if (uid_a, uid_b) in recent_matches: score += RECENT_MATCH_PENALTY`.
   - This makes re-matching the same opponent significantly less attractive, but still possible if no one else is available (the wait bonus will eventually overcome the penalty).

3. **Pass recent matches into the matchmaker:**
   - `run_matchmaking_wave()` currently takes only `queue`. Add an optional `recent_matches` parameter.
   - The orchestrator passes it from `StateManager`.

### Phase 2: Future тАФ Event-Driven / Faster Waves

If Phase 1 doesn't fully solve the problem:

1. **Reduce wave interval** from 60s to e.g. 15s or 10s. This alone may help тАФ shorter waves mean less "wasted" wait when a third player joins mid-cycle.

2. **Event-triggered matching:** When a player joins the queue, immediately check if any existing queue entry is a viable match. If so, create the match instantly without waiting for the next wave.
   - This requires moving matching logic into the queue-join endpoint or a triggered task.
   - Still use the wave as a fallback for expanding MMR windows over time.

3. **No-obligation waves:** The wave runs but doesn't have to produce a match. It only increments `wait_cycles` and expands windows. Actual matching happens opportunistically on queue join.

**Recommendation:** Implement Phase 1 first. It's backward-compatible, keeps the matchmaker stateless (recent matches are passed in, not stored internally), and directly addresses the repeat-opponent issue. Phase 2 can be layered on later if needed.

### Files to Modify (Phase 1)

- `backend/core/config.py` тАФ add `RECENT_MATCH_PENALTY_WINDOW`, `RECENT_MATCH_PENALTY`
- `backend/orchestrator/state.py` тАФ add `recent_matches_1v1` / `recent_matches_2v2` dicts
- `backend/orchestrator/transitions/_match.py` тАФ populate recent matches on match creation
- `backend/algorithms/matchmaker.py` тАФ accept and apply recent-match penalty in scoring
- `backend/algorithms/matchmaker_2v2.py` тАФ same for 2v2
- `tests/` тАФ add unit tests for penalty behavior

### Complexity

Phase 1: Medium. Phase 2: High. Recommend shipping Phase 1 and evaluating.

## 8. Add Discord Role on First Setup Completion

**Goal:** When a player completes `/setup` for the first time, assign them a Discord server role.

### Plan

1. **Config:**
   - New env var: `SETUP_COMPLETE_ROLE_ID` in bot config.
   - If unset, skip role assignment (same pattern as `CHANNEL_MANAGER_URL`).

2. **Where to assign the role:**
   - In the bot's setup completion callback (the final confirm in the setup flow).
   - After the backend confirms setup success, the bot calls Discord's role API.
   - This must happen **bot-side** because the bot holds the Discord gateway connection and guild member access.

3. **Implementation:**
   - In `bot/components/views.py`, in the setup confirm callback (where the backend `POST /commands/setup` succeeds):
     - Fetch the guild: `guild = bot.get_guild(GUILD_ID)` (need guild ID from config or from the interaction).
     - Fetch the member: `member = guild.get_member(interaction.user.id)` or `await guild.fetch_member(...)`.
     - Add the role: `await member.add_roles(discord.Object(id=SETUP_COMPLETE_ROLE_ID))`.
   - Wrap in try/except тАФ role assignment failure should NOT block setup completion. Log the error.

4. **"First time" check:**
   - The TODO says "first time." The backend already tracks `completed_setup_at`. If the player already had a setup timestamp before this `/setup` run, skip role assignment.
   - Alternatively, just always try to add the role тАФ Discord's `add_roles` is idempotent (no error if they already have it).

5. **Edge case тАФ DM-only commands:**
   - `/setup` runs in DMs. The bot needs guild access to assign roles.
   - `interaction.user` in a DM is a `User`, not a `Member`. Must fetch the member from the guild explicitly.
   - Requires the bot to have the guild cached or use `fetch_member`.

### Files to Modify

- `bot/core/config.py` тАФ add `SETUP_COMPLETE_ROLE_ID`
- `bot/components/views.py` тАФ add role assignment after setup confirm
- `.env.example` or docs тАФ document the new env var

### Complexity

Low. Main risk is the DM тЖТ guild member lookup, which requires the bot to have `GUILD_MEMBERS` intent or use `fetch_member`. The bot likely already has this for other features.
